# Phase 39/40 Assignment Recommendation and SLA Design

## Purpose

Phase 39 and Phase 40 add visible, defensible extension features on top of the existing V1 ticket workflow:

- Phase 39: recommend an agent for a pending ticket without automatically assigning it.
- Phase 40: calculate and display SLA deadline status from ticket priority.

The goal is not to build a full enterprise dispatch engine. The goal is to demonstrate that the current architecture can grow through clear service boundaries, explainable rules, and frontend evidence without breaking the manual testing flow that already works.

## Current Context

The project already has the key extension points:

- `TicketWorkflowService` owns ticket creation and workflow transitions.
- `AssignmentStrategy` and `ManualAssignmentStrategy` already exist.
- The `ticket` table already has `priority` and `deadline_at`.
- `Ticket` already carries `deadlineAt`.
- Frontend ticket list/detail pages already render status, priority, timestamps, assignment controls, and workflow actions.
- Admin ticket detail already supports manual assignment to active agents.

This phase should reuse those boundaries instead of introducing a parallel workflow.

### Existing TicketStatus Values

The implementation must use the existing `com.example.aiticket.ticket.domain.TicketStatus` enum exactly:

```text
PENDING_ASSIGN
PENDING_PROCESS
PROCESSING
RESOLVED
CLOSED
```

Status semantics for this phase:

- `PENDING_ASSIGN`: waiting for admin assignment.
- `PENDING_PROCESS`: assigned to an agent but not started.
- `PROCESSING`: actively being handled by an agent.
- `RESOLVED`: agent has marked the ticket resolved, but the creator has not confirmed closure yet.
- `CLOSED`: workflow is closed.

`RESOLVED` is intentionally treated differently by Phase 39 and Phase 40:

- Assignment recommendation counts `RESOLVED` as agent workload because it can still reopen and remains associated with the current agent until the user confirms closure.
- SLA treats `RESOLVED` as completed because the agent has delivered a solution within the SLA clock.

This difference is deliberate and must be preserved in tests and UI copy.

## Non-Goals

- No automatic assignment during ticket creation.
- No complex rule-table editor.
- No multi-department approval flow.
- No background escalation job.
- No notification system.
- No destructive data migration.
- No priority-edit workflow in this phase.
- No SLA deadline recalculation after creation. If priority changes are added in a later phase, that feature must explicitly decide whether to recalculate `deadline_at`.

## Phase 39: Assignment Recommendation

### Behavior

When an admin opens a `PENDING_ASSIGN` ticket detail page, the frontend requests an assignment recommendation.

The recommendation is visible but non-mutating:

1. Backend returns the recommended active agent.
2. Frontend shows the recommendation and reason.
3. Admin can click `使用推荐坐席` to fill the existing assignee select.
4. Admin still clicks the existing `分配给坐席` button to actually assign.

This preserves the current manual workflow while making the intelligent dispatch capability obvious in the UI.

### Recommendation Rule

The first implementation uses a deterministic load-based rule:

1. Candidate users must be active users with the `AGENT` role.
2. For each candidate, count currently active assigned workload.
3. Active assigned workload means tickets where:
   - `assignee_id = candidate.id`
   - `deleted = 0`
   - `status IN ('PENDING_PROCESS', 'PROCESSING', 'RESOLVED')`
4. Select the candidate with the lowest active ticket count.
5. If counts tie, select the lowest user ID for stable behavior.

The returned reason should be human-readable:

```text
推荐演示坐席：当前在办 2 单，是当前负载最低坐席
```

Reason text should be generated by one formatter so tests do not depend on duplicated string building. The initial formatter uses these cases:

- No active agent: `暂无可推荐坐席`
- One candidate: `推荐{displayName}：当前在办 {count} 单，是当前唯一可用坐席`
- Recommended count is 0: `推荐{displayName}：当前无在办工单，可优先承接`
- General case: `推荐{displayName}：当前在办 {count} 单，是当前负载最低坐席`

### API

Add:

```http
GET /api/tickets/{ticketId}/assignment-recommendation
```

Permission:

```text
ticket:assign
```

Response:

```json
{
  "recommendedAssigneeId": 3,
  "recommendedUsername": "agent",
  "recommendedDisplayName": "演示坐席",
  "activeTicketCount": 2,
  "reason": "推荐演示坐席：当前在办 2 单，是当前负载最低坐席"
}
```

If no active agent exists, return HTTP 200 with no recommendation:

```json
{
  "recommendedAssigneeId": null,
  "recommendedUsername": null,
  "recommendedDisplayName": null,
  "activeTicketCount": null,
  "reason": "暂无可推荐坐席"
}
```

If the ticket does not exist or is deleted, return the existing ticket-not-found behavior used by ticket detail APIs.

If the ticket exists but is not `PENDING_ASSIGN`, return HTTP 200 with no recommendation:

```json
{
  "recommendedAssigneeId": null,
  "recommendedUsername": null,
  "recommendedDisplayName": null,
  "activeTicketCount": null,
  "reason": "当前状态不需要分派"
}
```

### Backend Structure

Introduce a focused service for recommendation:

- `TicketAssignmentRecommendationService`
  - Loads the ticket and verifies it exists.
  - Returns an empty recommendation for non-`PENDING_ASSIGN` tickets.
  - Uses mapper query results to select the lowest-load active agent.

Introduce a small DTO/domain record:

- `AssignmentRecommendation`
  - `recommendedAssigneeId`
  - `recommendedUsername`
  - `recommendedDisplayName`
  - `activeTicketCount`
  - `reason`

Mapper additions should be small and read-only:

- list active agents
- count active assigned tickets by agent

The existing `AssignmentStrategy` remains part of the mutation path. The new recommendation service is read-only and feeds the UI.

### Frontend

On `TicketDetailView.vue`, only for admins with `ticket:assign` and `PENDING_ASSIGN` tickets:

- Show a `智能推荐` block above the existing assignee dropdown.
- Display recommended display name, username, current active ticket count, and reason.
- Add `使用推荐坐席` button.
- Clicking the button sets `selectedAssigneeId` to the recommended ID.
- The existing `分配给坐席` button remains the only assignment action.

If no recommendation exists, show `暂无可推荐坐席`.

## Phase 40: SLA

### Deadline Calculation

When creating a ticket from an AI session, calculate `deadlineAt` from priority:

```text
URGENT: createdAt + 4 hours
HIGH:   createdAt + 8 hours
NORMAL: createdAt + 24 hours
LOW:    createdAt + 72 hours
```

If no priority is supplied, `NORMAL` remains the default.

The calculated deadline is saved to existing `ticket.deadline_at`.

### SLA Status

Expose derived SLA fields in ticket list and detail responses:

- `deadlineAt`
- `slaStatus`
- `slaRemainingMinutes`

SLA status values:

```text
COMPLETED
OVERDUE
DUE_SOON
ON_TRACK
```

Rules:

1. If status is `RESOLVED` or `CLOSED`, SLA status is `COMPLETED` and remaining minutes is null.
2. If `deadlineAt` is missing, SLA status is `ON_TRACK` and remaining minutes is null.
3. If now is after `deadlineAt`, SLA status is `OVERDUE`.
4. If remaining minutes is less than or equal to 120, SLA status is `DUE_SOON`.
5. Otherwise SLA status is `ON_TRACK`.

Boundary behavior:

- `now == deadlineAt` is `DUE_SOON`, not `OVERDUE`.
- `remainingMinutes == 120` is `DUE_SOON`.
- `now > deadlineAt` by any positive amount is `OVERDUE`.

This is intentionally simple and deterministic. It is enough to demonstrate SLA awareness without requiring background jobs or escalation queues.

### Backend Structure

Introduce:

- `SlaPolicy`
  - Maps priority to due duration.
  - Calculates deadline for ticket creation.
- `SlaStatus`
  - Enum for `COMPLETED`, `OVERDUE`, `DUE_SOON`, `ON_TRACK`.
- `SlaSnapshot`
  - `deadlineAt`
  - `slaStatus`
  - `slaRemainingMinutes`
- `SlaCalculator`
  - Derives `SlaSnapshot` from ticket status, deadline, and current time.

`TicketWorkflowService#createFromAiSession` should use `SlaPolicy` to populate `deadlineAt`.

`TicketResponse` and `TicketDetailResponse` should include the derived SLA fields.

### Frontend

Update `frontend/src/api/tickets.ts`:

- Add `deadlineAt?: string | null`
- Add `slaStatus?: 'COMPLETED' | 'OVERDUE' | 'DUE_SOON' | 'ON_TRACK' | null`
- Add `slaRemainingMinutes?: number | null`

Frontend tests should not assert exact minute values from live time. They should assert status labels and use fixed mocked timestamps when testing remaining-time formatting itself.

Update ticket list:

- Add `SLA` column.
- Render SLA chip:
  - `ON_TRACK`: `正常`
  - `DUE_SOON`: `即将超时`
  - `OVERDUE`: `已超时`
  - `COMPLETED`: `已完成`
- Show deadline date under the chip when available.

Update ticket detail:

- Add right-side `SLA 状态` panel.
- Show:
  - SLA status label
  - deadline time
  - remaining time text
  - priority rule text

Remaining time text:

```text
已完成
已超时
剩余 1 小时 30 分钟
未设置截止时间
```

## Testing

Backend tests:

- `SlaPolicyTest`
  - Verifies each priority maps to the expected deadline offset.
- `SlaCalculatorTest`
  - Verifies completed, overdue, due-soon, on-track, and missing-deadline states.
  - Verifies `now == deadlineAt` is `DUE_SOON`.
  - Verifies `remainingMinutes == 120` is `DUE_SOON`.
  - Verifies `COMPLETED` remaining minutes is null.
- `TicketWorkflowServiceTest`
  - Verifies created tickets persist a deadline based on priority.
- `TicketAssignmentRecommendationServiceTest`
  - Verifies lowest-load active agent is selected.
  - Verifies stable tie-break by user ID.
  - Verifies empty recommendation when no active agents exist.
  - Verifies ticket-not-found behavior.
  - Verifies non-`PENDING_ASSIGN` tickets return the `当前状态不需要分派` empty recommendation.
  - Verifies reason formatting for one candidate, zero workload, and general lowest-load cases.
- `TicketControllerTest`
  - Verifies recommendation endpoint permission annotation and response shape.

Frontend tests:

- `tickets.spec.ts`
  - Verifies recommendation API path.
- `TicketListView.spec.ts`
  - Verifies SLA column/chip/deadline rendering.
  - Avoids brittle assertions on exact live remaining-minute values.
- `TicketDetailView.spec.ts`
  - Verifies recommendation block appears for pending admin tickets.
  - Verifies `使用推荐坐席` fills the existing assignee select.
  - Verifies SLA panel renders deadline and remaining time.

Runtime verification:

- Existing Phase 31 acceptance evidence script must still pass.
- Browser manual check should cover:
  1. Admin opens a pending ticket detail.
  2. Recommendation block appears.
  3. `使用推荐坐席` fills the assignee dropdown.
  4. Admin assigns the ticket normally.
  5. Ticket list shows SLA chip.
  6. Ticket detail shows SLA panel.

## Documentation Impact

The project plan and final README should describe these as implemented extension features:

- Strategy-based assignment recommendation.
- Priority-based SLA deadline and frontend SLA visibility.

The thesis/PPT can present them as proof that the system is not only a CRUD workflow, but has extensible service boundaries for operations management.
