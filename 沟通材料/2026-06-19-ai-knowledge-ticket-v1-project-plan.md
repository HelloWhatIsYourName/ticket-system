# 面向企业服务场景的 AI 知识库问答与工单协同处理平台第一版项目计划书

## 1. 项目名称

面向企业服务场景的 AI 知识库问答与工单协同处理平台设计与实现。

## 2. 项目背景

在企业售后服务、内部 IT 支持、产品咨询、运维报修等场景中，用户问题通常具有重复率高、响应时效要求高、处理流程依赖人工经验等特点。传统工单系统主要承担记录和流转职责，缺少对知识库内容的智能利用，也难以在用户提交问题的第一时间完成语义理解、自动解答、问题分类和处理建议生成。

随着大语言模型、向量检索和企业知识库技术的发展，企业服务平台可以从“被动记录工单”升级为“优先智能解答，必要时协同处理”的服务模式。本项目拟构建一个结合 AI 知识库问答、工单协同处理、基于角色的访问控制和实时数据能力的企业服务平台，以提高问题响应效率、降低人工坐席重复劳动，并增强服务过程的可追踪性和可管理性。

## 3. 项目目标

本项目第一版目标是实现一个具备完整业务闭环的企业服务平台。系统应支持管理员维护企业知识库，用户通过自然语言提问获得 AI 回答；当 AI 无法给出可靠解答或用户仍需人工处理时，系统可自动生成工单并进入坐席处理流程。平台同时提供角色权限控制、工单状态流转、知识库管理和基础统计能力。

第一版重点不是实现复杂审批流和完整 SLA 管理，而是搭建稳定、清晰、可扩展的核心架构，为后续增加多部门流转、SLA 超时预警、审批节点、数据权限和更复杂调度策略预留空间。

## 4. 建设范围

### 4.1 第一版纳入范围

1. 普通用户注册、登录、权限识别和基于角色的访问控制。
2. 管理员上传和维护知识库文档。
3. 系统解析文档、切分知识片段并生成向量索引。
4. 用户通过 AI 问答入口提交自然语言问题。
5. 系统基于知识库检索结果进行 RAG 问答。
6. 对无法解决的问题支持自动转工单。
7. 坐席查看、接收、处理和标记解决工单。
8. 工单状态流转和流转记录追踪。
9. Redis 支撑热门问答缓存、接口限流、短期上下文缓存和异步处理队列。
10. 管理端提供知识库、用户、角色、权限、工单分类和基础统计管理。

### 4.2 第一版暂不实现范围

1. 复杂多部门审批流。
2. 完整 SLA 规则引擎和超时升级机制。
3. 多租户企业隔离。
4. 复杂组织架构和细粒度数据权限。
5. 专业客服 IM 系统。
6. 多模型自动路由和本地大模型部署。

上述能力不在第一版实现，但会在数据结构、服务边界和流程设计中预留扩展点。

## 5. 用户角色与权限设计

系统采用 RBAC 模型进行权限控制，第一版包含四类角色。

### 5.1 普通用户

普通用户主要使用 AI 问答和工单提交能力。其权限包括发起 AI 问答、查看问答历史、将问题转为工单、补充工单信息、查看本人提交的工单进度以及对已解决工单进行评价。

### 5.2 坐席或工程师

坐席或工程师负责人工处理工单。其权限包括查看分配给自己的工单、接收或处理工单、查看 AI 生成的问题摘要和关联知识片段、回复用户、更新工单状态以及标记工单已解决。

### 5.3 管理员

管理员负责平台业务配置和运营管理。其权限包括管理知识库文档、维护工单分类、管理普通用户和坐席账号、分配角色权限、查看工单统计和知识库命中情况。

### 5.4 超级管理员

超级管理员负责系统级配置。其权限包括管理全部角色、权限、菜单、模型配置、系统参数和审计日志。第一版中超级管理员可以与管理员共用部分界面，但权限边界应在数据模型和接口层面保留。

## 6. 功能性需求

### 6.1 认证与权限模块

系统应支持用户登录、登出、JWT 鉴权、角色识别和接口权限校验。后端通过 Spring Security 实现认证授权，关键接口使用方法级权限控制。前端根据用户权限动态展示菜单和操作按钮。

权限模型包括用户、角色、权限、菜单、用户角色关系和角色权限关系。第一版实现菜单级和接口级权限控制，关键操作按钮根据权限进行前端显示控制，不实现复杂数据范围权限。

普通用户可以自助注册并默认获得普通用户角色，坐席、管理员和超级管理员账号由管理员或超级管理员在后台创建，不开放自助注册。

### 6.2 知识库管理模块

管理员可以上传企业制度、产品手册、FAQ、操作指南和历史处理文档。系统应记录文档名称、分类、版本、上传人、状态、解析状态和创建时间。

系统通过 Apache Tika 对文档内容进行解析，并将文本按合理长度切分为知识片段。文档切片采用段落优先、长度兜底的策略：优先按标题和自然段落切分；若单个段落过长，则按固定字符长度继续切分；相邻片段保留一定重叠窗口，减少上下文断裂问题。第一版可将单个片段控制在 500 至 800 字左右，重叠窗口控制在 80 至 120 字左右，具体值根据 Embedding 模型上下文长度和测试效果调整。

每个知识片段应保留来源文档、段落内容、片段序号、向量值和元数据，便于后续检索和答案溯源。元数据至少包括 `chunk_index`、`source_title`、`source_page`、`document_id`、`category_id` 和文档启用状态。

知识库文档需要具备明确的解析状态机：待解析、解析中、解析成功、解析失败。解析失败时记录失败原因和重试次数，管理员可触发重试。第一版将解析重试次数限制为 3 次，超过后标记为解析失败并保留人工重试入口。

第一版 Embedding 采用硅基流动 `Qwen/Qwen3-Embedding-8B`，向量维度按 1024 维配置，Oracle 中知识片段向量字段设计为 `VECTOR(1024, FLOAT32)`。第一版演示数据规模预计为数百到数千个知识片段，优先采用精确向量检索，即通过 `VECTOR_DISTANCE(embedding, :queryVector, COSINE)` 按余弦距离全表排序召回 top-k 片段，暂不引入 HNSW 或 IVF 近似向量索引，以降低 Oracle 23ai Free 版本适配风险。

### 6.3 AI 问答模块

用户输入问题后，系统首先进行基础校验和限流，然后对问题生成向量表示，并在 Oracle 23ai 中进行向量相似度检索。第一版默认召回 Top-K 个相似知识片段，K 初始设置为 5，并支持通过配置调整。检索时只召回已启用且解析成功的文档片段；如果用户问题已经明确分类，可优先在对应分类下检索，分类结果不可靠时退回全局检索。

系统将检索到的相关知识片段与用户问题组合为提示词，请求大模型生成回答。回答结果应返回引用片段编号和来源信息，便于用户查看答案依据，也便于坐席在转工单后追踪 AI 判断来源。

提示词需要明确约束模型行为：只能基于召回的知识片段作答；知识库没有覆盖的信息应说明无法确认；不得编造制度、流程、价格、时间、联系方式或处理结论；引用来源必须来自本次检索片段。若召回片段不足以回答问题，模型应通过结构化字段声明 `can_answer=false`，并给出转人工原因。

AI 回答应包含答案正文、引用知识片段、置信度或可靠性提示。第一版对“无法可靠回答”的判断采用工程近似策略，而不将其表述为严格概率判断。判定依据包括三类：

1. 检索分数阈值：top-k 知识片段最高相似度低于阈值时，判定为知识库覆盖不足。阈值初始可设置为 0.70，并在测试集上微调。
2. 有效召回数量：满足最低相似度要求的知识片段数量为 0，或低于系统配置的最小召回数量时，提示转人工。
3. 模型结构化自评：提示词要求大模型输出 `can_answer`、`confidence` 和 `reason` 等结构化字段。当模型声明无法依据知识库可靠回答时，系统提示转人工。

用户始终保留手动转人工入口。系统所谓置信度主要来自检索相似度、有效召回数量和模型自评结果的组合，不等同于统计学意义上的真实概率。

### 6.4 自动转工单模块

当用户主动选择转人工，或系统判断 AI 回答无法解决问题时，平台应支持自动生成工单。生成工单时，大模型可辅助提取问题摘要、问题分类、优先级辅助建议和处理建议，最终优先级由管理员或坐席确认。

工单应关联原始问答会话、用户问题、AI 回答、命中的知识片段和转人工原因，以便坐席理解上下文。

### 6.5 工单协同处理模块

第一版工单流程包括待分配、待处理、处理中、已解决和已关闭五种核心状态。坐席可以查看分配给自己的工单，阅读 AI 摘要和知识库依据，填写处理回复并更新状态。

系统应将每次状态变化、处理意见、操作人和操作时间记录到工单流转日志中。工单主表只保存当前状态，完整过程由流转记录表保留。

第一版状态迁移规则如下：

| 当前状态 | 触发动作 | 操作者 | 目标状态 |
| --- | --- | --- | --- |
| 待分配 | 分配坐席 | 管理员 | 待处理 |
| 待分配 | 取消或关闭无效工单 | 管理员 | 已关闭 |
| 待处理 | 开始处理 | 坐席 | 处理中 |
| 待处理 | 关闭无效工单 | 管理员 | 已关闭 |
| 处理中 | 标记解决 | 坐席 | 已解决 |
| 处理中 | 关闭无效工单 | 管理员 | 已关闭 |
| 已解决 | 反馈未解决 | 普通用户 | 处理中 |
| 已解决 | 确认并评价 | 普通用户 | 已关闭 |
| 已解决 | 超时未确认 | 系统 | 已关闭 |

状态编码建议采用 `PENDING_ASSIGN`、`PENDING_PROCESS`、`PROCESSING`、`RESOLVED` 和 `CLOSED`。第一版支持轻量级重新处理：用户在已解决状态反馈未解决时，工单回到处理中，不单独增加 `REOPENED` 状态；重新处理次数、原因和操作人记录在 `ticket_flow_log` 中。这样可以满足基础闭环，又不会提前引入完整审批工作流。

### 6.6 Redis 实时能力模块

Redis 在第一版中承担以下职责：

1. 缓存热门问题和高频问答结果。
2. 缓存用户登录状态、权限摘要或短期会话数据。
3. 对 AI 问答接口和登录接口进行限流。
4. 使用 Redis Stream + Consumer Group 承载文档解析、向量生成等异步任务。
5. 可缓存用户 AI 短期会话上下文，完整问答历史以 Oracle 为准。
6. 使用 ZSet 统计热门问题、知识库命中次数和工单分类热度。

Redis 任务队列必须使用确认机制。文档解析和向量生成任务通过 Redis Stream 投递，消费者处理成功后执行 ACK；消费者异常退出时，任务保留在 pending 列表中，由恢复任务重新投递。任务失败时更新 Oracle 中的解析状态和失败原因，超过重试上限后进入失败状态，不再无限重试。

### 6.7 统计与管理模块

管理端提供基础统计能力，包括工单总量、待处理工单数、已解决工单数、平均处理时长、知识库命中率、热门问题排行和高频分类分布。统计图表使用 ECharts 展示。第一版统计以基础运营看板为目标，不实现复杂 BI 分析。

第一版统计指标计算口径如下：

| 指标 | 计算口径 |
| --- | --- |
| 工单总量 | 指定时间范围内创建的工单数量 |
| 待处理工单数 | 状态为待分配、待处理或处理中的工单数量 |
| 已解决工单数 | 状态为已解决或已关闭的工单数量 |
| 平均处理时长 | 从工单创建时间到首次标记已解决时间的平均耗时；重新处理工单仍按首次标记已解决时间计算，重开后的追加耗时不计入第一版平均处理时长 |
| 知识库命中率 | 从 `ai_message_citation.similarity` 聚合每条 AI 回答的最高引用相似度，`similarity >= 0.70` 的回答次数 / 有引用相似度的回答次数，用于反映知识库覆盖情况，不等同于回答质量 |
| 热门问题排行 | 按归一化后的问题文本或关键词在 Redis ZSet 中累计次数 |
| 高频分类分布 | 按工单分类统计指定时间范围内的工单数量 |

热门问题排行第一版不做语义聚类，仅进行空格、标点、大小写等基础归一化，或按提取关键词进行简化统计。语义相似问题合并归类作为后续扩展能力。

## 7. 非功能性需求

### 7.1 安全性

系统应对所有需要登录的接口进行鉴权，关键管理接口应进行权限校验。用户密码需要加密存储。AI 提问、文档上传和工单回复等输入应进行长度限制、格式校验和基础关键词过滤。第一版不接入专业内容审核服务。

AI 场景需要额外考虑 Prompt Injection 风险。系统提示词应明确要求模型只根据本次检索到的授权知识片段回答，不执行用户要求改变系统规则、泄露系统提示词、输出隐藏配置或访问未授权信息的指令。若用户问题或知识片段中出现试图覆盖系统规则的内容，应作为普通文本处理，不改变系统约束。

知识库检索需要遵守基础权限和状态过滤。第一版知识库文档默认对所有登录用户可检索，管理员可控制文档启用状态；未启用、解析失败、删除状态或未通过基础校验的文档不得进入检索结果。

文件上传需要进行大小限制、类型白名单、文件名安全处理和上传目录隔离。系统只允许上传第一版支持的文档类型，保存时使用服务端生成的安全文件名，避免直接使用用户提交的原始文件名作为存储路径。解析失败需要记录失败原因并进入可重试状态，不得影响其他文档解析任务。

### 7.2 可扩展性

系统应将 AI 服务、知识库服务、工单工作流服务、权限服务和 Redis 能力封装为相对独立的模块。工单状态变更必须通过统一的工作流服务完成，避免状态逻辑散落在控制器中。

工单分派策略应通过策略接口封装。第一版只采用管理员手动分配，系统根据 AI 分类结果、工单分类和坐席信息给出推荐分类或推荐坐席，但不自动决定最终分派。这样可以避免第一版引入坐席负载、在线状态和技能匹配等复杂规则，同时保留后续扩展空间。

第一版策略接口可设计为 `AssignmentStrategy`，核心方法为 `selectAssignee(TicketCreateContext context)`。第一版仅提供 `ManualAssignmentStrategy` 实现，由管理员确认最终处理人；后续自动分派能力统一放入第 11 章扩展清单。

### 7.3 可维护性

数据库结构通过 Flyway 管理版本。后端采用分层架构，控制器层只负责参数接收和响应封装，业务规则集中在 Service 层，复杂 SQL 可放入 MyBatis XML。前端按角色和业务模块拆分页面。

### 7.4 性能要求

普通业务接口在正常数据量下应保持稳定响应。AI 问答由于依赖外部模型服务，第一版采用 SSE 流式返回降低用户等待感，并保留普通 HTTP 响应作为异常降级方案。热门问答、权限摘要和统计排行使用 Redis 缓存，避免频繁查询数据库。

### 7.5 可审计性

系统应记录关键操作日志，包括登录、知识库上传、角色权限变更、工单状态变更和模型配置变更。第一版可先实现基础审计日志表，后续再扩展为完整审计查询模块。

### 7.6 测试与效果评估

除常规接口测试和流程联调外，系统需要建立轻量级 RAG 效果评估集。第一版准备 20 至 30 条企业服务场景问题，人工标注期望命中的知识片段和参考答案，用于统计检索命中率、回答有用率和误转工单率。该评估不追求学术级严密性，但应为论文和答辩提供可量化依据。

## 8. 技术栈方案

### 8.1 后端技术

后端采用 Spring Boot 3 和 Java 21。Spring Security 用于认证授权，JWT 用于无状态身份凭证。MyBatis-Plus 负责基础 CRUD，MyBatis XML 负责复杂查询和 Oracle 特定 SQL。Flyway 负责数据库迁移。Oracle `VECTOR` 字段的写入、读取和相似度检索不依赖 MyBatis-Plus 自动 CRUD，统一通过 MyBatis XML 和必要的自定义 TypeHandler 实现。

### 8.2 数据库与缓存

主数据库采用 Oracle Database 23ai Free，并通过 Docker Compose 部署。Oracle 负责存储用户、角色、权限、知识库文档、知识片段、向量数据、问答会话、工单、流转记录和审计日志。第一版在正式开发知识库模块前必须完成 Oracle 向量最小闭环验证，包括创建 `VECTOR(1024, FLOAT32)` 字段、写入硅基流动向量、通过 `VECTOR_DISTANCE(embedding, :queryVector, COSINE)` 检索 top-k 片段、验证 MyBatis 绑定向量参数并通过 Spring Boot 查询返回结果。若该验证不可接受，应及时评估 PostgreSQL + pgvector 作为备选方案。

Redis 7 负责缓存、限流、短期上下文、热门排行和异步队列。Redis 不作为核心业务数据的最终存储，关键业务状态、完整问答历史和工单状态仍以 Oracle 为准。异步队列采用 Redis Stream + Consumer Group，不使用简单 List 队列承载需要可靠处理的任务。

### 8.3 AI 与文档处理

Chat 模型和 Embedding 模型使用独立供应商配置。第一版 Chat 采用 DeepSeek API，Embedding 采用硅基流动 `Qwen/Qwen3-Embedding-8B`，分别配置 `ai.chat.*` 和 `ai.embedding.*` 的 `baseUrl`、`apiKey`、`model` 和向量维度。Apache Tika 用于解析 PDF、Word、TXT、Markdown 等文档。

### 8.4 前端技术

前端采用 Vue3、TypeScript、Element Plus 和 ECharts。普通用户侧重点是问答和工单进度，坐席侧重点是工单处理工作台，管理员侧重点是知识库管理、权限管理和统计图表。

### 8.5 部署方式

开发和演示环境使用 Docker Compose 管理 Oracle、Redis 和后端相关依赖。第一版至少提供一套可复现的本地启动方案，降低答辩演示风险。

## 10. 核心业务流程

### 10.1 知识库入库流程

管理员上传文档后，系统保存文档元数据和原始文件，并将文档状态置为待解析。随后生成 Redis Stream 异步解析任务，消费者领取任务后将状态更新为解析中。解析阶段先提取标题、页码和正文，再按段落优先、长度兜底、相邻重叠的策略切分知识片段，并写入 `chunk_index`、`source_page`、`source_title` 等元数据。系统按批次调用硅基流动 Embedding API 生成向量，将片段内容、向量、来源文档和元数据写入 Oracle，避免逐条请求导致耗时过长或触发限流。解析完成后更新文档状态为解析成功；失败时记录失败原因和重试次数，超过上限后标记为解析失败。

### 10.2 AI 问答流程

用户提交问题后，系统检查登录状态和接口限流。随后生成问题向量，在 Oracle 中检索相似知识片段。检索默认返回 Top-5 片段，并过滤未启用或解析失败的文档；存在可靠分类时优先按分类过滤。系统将检索结果、引用编号、来源信息和用户问题构造成提示词，请求大模型生成回答，并要求模型遵守“仅基于召回片段回答、无法确认则说明不知道、不得编造来源”的约束。系统将回答、引用片段、可靠性判断和会话记录持久化。第一版以单轮问答为主，可将最近若干轮对话作为补充上下文简单拼接；暂不实现复杂查询改写、指代消解和长期记忆。

### 10.3 自动转工单流程

当用户选择转人工或系统判断问题未解决时，系统调用大模型生成问题摘要、分类、优先级辅助建议和处理建议。后端创建工单并关联原问答会话，同时写入工单流转记录。坐席进入工作台后可看到该工单及 AI 提供的处理上下文，管理员或坐席确认最终优先级。

### 10.4 工单处理流程

坐席查看工单后，可以填写处理意见、引用知识库内容或使用 AI 生成回复建议。工单状态先从待分配进入待处理，再由坐席开始处理后进入处理中；问题解决后标记为已解决，用户确认、超时未确认或管理员关闭无效工单后进入已关闭状态。若用户反馈未解决，工单从已解决回到处理中，并在流转日志中记录重新处理原因。

## 13. 风险与应对

### 13.1 Oracle 23ai 向量能力学习成本

Oracle 23ai 的向量检索能力相对新，可能存在语法、驱动和 ORM 适配问题。应对策略是在第二阶段设置硬性技术 Spike，尽早验证最小闭环，即插入向量、检索相似片段和通过 Spring Boot 返回结果；复杂向量 SQL 使用 MyBatis XML 和自定义 TypeHandler 编写。若 Spike 失败或实现成本明显超出毕业设计周期，应评估 PostgreSQL + pgvector 作为备选方案。

### 13.2 大模型 API 稳定性风险

大模型 API 和 Embedding API 可能存在延迟、限额或不可用情况。应对策略是分别封装 Chat 客户端和 Embedding 客户端，保留模型供应商切换能力；同时对问答结果做持久化和缓存，对文档切片采用批量 Embedding 和失败重试，降低重复请求。

### 13.3 第一版范围膨胀风险

工单系统容易扩展出审批、SLA、IM、组织架构和多租户等复杂能力。第一版应严格围绕 AI 问答、自动转工单、坐席处理、知识库管理和 RBAC 展开，其他能力只预留扩展点。

### 13.4 文档解析质量风险

不同格式文档解析效果可能不一致。第一版应优先支持 PDF、DOCX、TXT 和 Markdown，上传文档应准备格式规范的演示材料，保证答辩演示稳定。

## 14. 第一版验收标准

1. 用户能够登录系统，并根据角色看到不同菜单和操作权限。
2. 管理员能够上传知识库文档，系统能够解析并生成可检索知识片段。
3. 用户能够通过自然语言提问，并获得基于知识库内容的 AI 回答。
4. AI 回答能够展示引用来源或关联知识片段。
5. 用户能够将未解决问题转为工单。
6. 坐席能够查看和处理分配给自己的工单。
7. 工单状态变化能够被记录和追踪。
8. 管理员能够查看知识库、用户权限和基础工单统计。
9. Redis 在缓存、限流、队列或排行中至少有三类明确使用场景。
10. 系统结构能够支持后续扩展多部门审批、SLA 和智能分派。
11. 系统能够根据检索相似度、有效召回数量、模型结构化自评和用户手动操作完成转人工判定。
12. RAG 评估集能够输出检索命中率、回答有用率和误转工单率等基础指标。
13. AI 问答能够通过 SSE 流式返回，并在异常场景下保留普通 HTTP 响应降级方案。
14. 文件上传和知识库检索能够执行类型白名单、大小限制、文档状态过滤和基础 Prompt Injection 防护。

## 15. 初步结论

本项目以企业服务场景为背景，将 AI 知识库问答与工单协同处理结合起来，形成从用户提问、自助解答、自动转工单到坐席处理的完整闭环。技术上采用 Spring Boot 3、Oracle 23ai、Redis 7、Vue3、DeepSeek Chat API 和硅基流动 Embedding API，既具有工程可实现性，也具备较强的新颖性和答辩展示价值。

第一版应优先保证核心链路跑通和架构边界清晰，在不实现复杂多部门审批和 SLA 的前提下，通过字段预留、服务抽象和流转日志设计，为后续功能扩展保留空间。
